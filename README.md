# Симулятор турнира математических боев с возможностью трансляции текущего турнира

## Заказчик 

* [Казанский Математический Центр](https://anokmc.ru)
* [Правила проведения турнира](https://github.com/BikTaIl/IPS_HSE_Bikmukhametov_Tagir_2026/blob/main/images/25г%2009%20правила%20флеш-боя.pdf)
___

## Верхнеуровневое описание проекта

Веб-приложение, позволяющее:

* задать **силу команд** (в терминах матбоев - результаты командной олимпиады).
* **составить турнирную сетку**.
* **смоделировать проведение турнира** множество раз.
* визуализировать распределение возможных итоговых мест и исходов матчей.
* транслировать результаты текущего турнира.
___

## Цели проекта
  1. Первоначальной целью ставится удовлетворение запроса заказчика: возможность прямой трансляции результатов турнира, а также удобство пользования приложением для администраторов.
  2. Предоставить свободному пользователю возможность смоделировать собственные турниры, их результаты при различных вводных данных.
  3. Исследовать влияние командной (распределительной) олимпиады, турнирной сетки и количества лиг (групп) на результаты всего турнира математических боев.
___
## Предварительное исследование

Необходимо

  1. Выгрузить имеющиеся данные по результатам турниров.
  2. Выявить зависимость вероятности победы от изначального распределения сил.

Для исследования будут использованы ресурсы нейронных сетей в целях парсинга данных. Библиотека Pandas для визуализации данных. Предварительная гипотеза: **можно использовать методику, которая применяется в шахматах**

## Возможности создаваемого приложения.
Приложение представляет собой 4 вкладки.
  1. Вкладка администратора.
  2. Вкладка-конструктор.
  3. Вкладка для трансляции текущего турнира.
  4. Вкладка с общей информацией о самом турнире (свод правил, ЧаВО).

Внизу каждой страницы основная информация об организации (наименование ЮЛ, контактные данные)
Возможности каждой вкладки описаны ниже
### Вкладка администратора

Это вкладка, на которой виден список всех администраторов сайта. При ее открытии требует залогиниться/зарегистрироваться, если пользователь зашел без аккаунта.
На вкладке можно:
  1. Добавить нового администратора по логину (только если такой логин существует).
  2. Удалить добавленного ранее администратора (или любого администратора, который находится строго ниже по иерархии. Здесь "строго ниже" означает, что права были выданы администратором, который получил права от пользователя, или же ниже по цепочке вниз).
  3. Аналогичные действия с ролью "редактор" - удаление и добавление. Редактор - более низкая роль, ее пользователя может лишить любой администратор вне зависимости от того, кто роль выдал.
  4. Администратор может создавать новых пользователей для выдачи логинов

### Вкладка-конструктор

Здесь можно смоделировать свой турнир. Вводные можно задать тремя способами:
  1. Занести собственный csv в определенном формате.
  2. Самостоятельно завести данные (задать количество команд, силы каждой).
  3. Воспользоваться встроенным рандомайзером (вручную заведя количество команд и далее по нажатию кнопки случайным образом раздать "силы").
После можно выбрать
  1. Количество групп (и задать количество команд в каждой группе). Не более 100, дабы не перегружать мощности. Если симуляций 100, то $n^2 \cdot 100$ - примерная асимптотика прогона 100 турниров. $10^6$ операций - логичная верхняя грань. 
  2. Турнирную систему (швейцарская, круговой турнир, на вылет (новые могут добавиться в будущем)). По умолчанию швейцарская, как на самом турнире.
  3. Собственную формулу вероятности победы команды в зависимости от соотношения сил (функция от m, n, p, k в формате Python). По умолчанию будет написана полученная при исследовании).
Далее запускается симуляция из 100 турниров. Выводится таблица: для каждой команды ее название, лига и среднее место по итогам 100 турниров.

### Вкладка для трансляции текущего турнира.

У вкладки в оглавлении название и номер турнира большим шрифтом. Под ними мелким город проведения (Казань) и год проведения. Ниже общая информация о турнире ("объединяет множество городов, проводится много лет"). Слева, кроме кнопки "главная", панель с кнопками по классам, за которыми можно понаблюдать. На каждой панели класса есть
  1. Общая сводка по классу (Квадратная таблица с результатами. Пример на картинке).
  ![Сводка ЛО](https://github.com/BikTaIl/IPS_HSE_Bikmukhametov_Tagir_2026/blob/main/images/Пример%20сводки%20ЛО.jpg)
  2. Текущее расписание боев (номер тура, команды, судьи, место и время проведения боя, место и время проведения разбора для жюри). Счет боя (либо надпись "Не окончен").
  3. Результаты командной олимпиады.

Кроме данных общих кнопок, администраторам открывается кнопка "редактирование списка команд". На открывшейся панели имеется таблица с текущим списком команд в формате
"рабочее название, детское название, класс". Лига может быть не определена. Отдельную команду можно удалить, также можно добавить команду по кнопке (откроется диалоговое окно для заполнения). Можно загрузить список команд файлом csv. **Важно:** список классов будет определяться автоматически по данным с этой панели.

Также администраторы и редакторы видят на данной вкладке кнопку "добавление результатов боя". Там видна таблица с введенными результатами боев вида (Команда1, Команда2, класс, тур, очки команды1, очки команды2). Есть кнопка "добавить результат", по которой открывается диалоговое окно. Сначала выбираются класс и лига (без этого не будет работать всплывающий список), далее из всплывающего списка выбираются команды, вводится счет. Также можно удалить в таблице результат любого добавленного пользователем боя (чужие нельзя).

Администраторы также видят кнопку "добавление результатов командной олимпиады" с таблицей результатов командной олимпиады (класс, название команды, номера задач, баллы за них, количество сданных задач, количество минусов, лига). Можно как одиночно добавить строку для команды, так и загрузить CSV. 

Суперадминистратор (все особенности пользователя будут описаны ниже) имеет возможность добавить новый турнир. При нажатии на эту кнопку необходимо произвести дополнительную проверку путем введения пароля суперадминистратора. Все данные предыдущего турнира будут стерты. Администратор же должен будет заполнить основные данные нового турнира (номер, время и место проведения). Функционал может быть изменен в рамках дальнейшего взаимодействия с заказчиком (вне рамок ИПС), в виду ограниченности временных ресурсов задача архивирования результатов остается на прежних инструментах.

### Вкладка с общей информацией о турнире

Здесь лежат документы с основной информацией о турнире: свод правил, информационное письмо, ответы на частые вопросы. Администратор имеет возможность загружать файлы в данные поля (ровно один файл для свода правил и для информационного письма). Администратор также может добавить ответ на вопрос по кнопке (выйдет диалоговое окно с требованием ввести текст вопроса и текст ответа на него). 

### Роли
  1. Пользователь - обыкновенный пользователь. Права описаны выше
  2. Редактор - пользователь с правами добавления результатов боя. Бизнес-суть - роль выдается стажерам на турнире для заполнения результатов.
  3. Администратор - пользователь с расширенными правами. Бизнес-суть - роль выдается членам оргкомитета и другим доверенным лицам, которым необходима более глубокая работа с турнирной таблицей: добавление новых команд, информации о турнире.
  4. Суперадминистратор - наделенный самым широким доступом пользователь. Роль нельзя выдать, можно только залогиниться под его аккаунтом. Создается разработчиком и позволяет создать новый турнир + наделен правами администратора. Бизнес-суть - доступы к логину и паролю должны иметь только главы оргкомитета, главы организации.

## Техническая информация
  1. Бэкенд будет написан при помощи Python.
  2. Фронт реализован на Java Script.
  3. БД развернута на базе Postgre. Работа с SQl производится при помощи встроенной ORM Django.
  4. Проект будет жить в docker.
  5. При разработке проект разворачивается на локальной машине. По окончании разработки хостинг проекта происходит на стороне виртуальной машины/сервера заказчика.
___
## Требования к коду 
Требования написаны исполнителем для исполнителя для самоконтроля
  1. Покрытие тестами не менее 65% кода
  2. Чистая архитектура (соответствие SOLID)
  3. Чистый код (использование типизации и комментариев для реализуемых функций), читаемое расставление отступов.
